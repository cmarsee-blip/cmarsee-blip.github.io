Why I rebuilt my personal site — again

This is the third time I've rebuilt this site. The first version was a single HTML file I wrote in a weekend and never touched again. The second was an over-engineered React app with a CMS I built myself, for an audience of exactly one visitor a month.

This time I gave myself one rule: nothing here should need a build step. If I can't fix a typo by editing a file and pushing it, I won't fix the typo, and eventually I'll stop writing.

What actually mattered

Looking back at the sites I kept updating versus the ones I abandoned, the difference was never the tech stack. It was how many steps stood between "I have a thought" and "that thought is published." Every extra step — a build pipeline, a CMS login, a deploy command I'd forget — was a chance to just not bother.

The best publishing tool is the one that's still open in your editor tomorrow.

So this version is plain HTML, CSS, and a sprinkle of JavaScript, hosted on GitHub Pages. Writing a new post means duplicating a file. Publishing means a git push. There's no dashboard to log into and no dependency to update six months from now.

What I'd tell past me

Pick the boring option first. You can always add complexity later if the simple version is actually holding you back — but for a personal site, it rarely is. The interesting part was never the framework. It was having somewhere to put the things I'd built.
