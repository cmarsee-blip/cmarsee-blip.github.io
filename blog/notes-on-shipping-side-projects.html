Notes on shipping side projects

Most of my side projects die in the same place: the point where it works on my machine and I have to decide whether it's worth the extra hour to make it work anywhere else.

That hour is boring. It's writing a README, checking the thing doesn't explode with an empty database, and deciding where it's actually going to live. None of it is the fun part, which is exactly why it gets skipped, and why so many projects stay a folder on one laptop forever.

The habits that changed this for me

I start every project with a live deploy target before I've written the feature I actually care about — even if it just renders a blank page. Once something is public, finishing it feels like tidying up rather than a whole separate project.

I also write the README first, as a kind of spec. If I can't describe what the thing does in three sentences before I've built it, I'm usually still figuring out what I'm making, and that's worth knowing early.

What I stopped doing

I stopped aiming for "impressive" launches. Almost everything on this site shipped quietly, to nobody, and picked up a handful of users later because it solved a real problem well. The projects I over-polished before showing anyone are, without exception, the ones still sitting unfinished.
